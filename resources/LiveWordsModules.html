LiveWords Modules Architecture

Basic Idea / Vision

  • Highlight interesting terms in text
  • Deliver related information and functionality “on the spot,” via a dynamic user interface
  • Extensible architecture allowing publishers to write their own modules
  • Make it easy for publishers to use LiveWords to relate to other content on Yahoo or their own network
  • create a gallery of modules that provide tailored functionality. Examples are the multimedia module for music entities, map modules for geo entities, a Search module for enabling related search., etc.

Design

The fundamental idea is to make it very easy to use and extend livewords. For that we need

  1. An easy way to enable the Livewords platform to detect custom sets of entities (i.e., Yahoo! News has its own sets of terms to be highlighted), and
  2. An even easier way of hooking new display modules into the overlay.

Publishers can have the choice of annotaing their content during publish time or during view-time.

Custom dictionaries could be provided in an XML form as described here.

Check out the architecture overview presentation (powerpoint): livewords_modules.ppt

Implementation

  • LiveWords Javascript Framework -- a single javascript file that needs to be included by a publisher. This script automatically detects on whether the current document requires view-time annotation or already is annotated. In either case, the script does setup the appropriate Livewords modules per annotation. Doing view-time annotation potentially requires requesting data from an service. The most sutable techniques for this would be XMLHttpRequest? (intra-domain) or JSON scripting (cross-domain requests).

  • A configuration mechanism which allows each publisher to define which entities should be detected and which modules should be used. In it's simplest form this configuration is either included in the document or given through a URL (i.e., on the publishers network).

  • Support for external livewords modules by (a) using a generic module that will access an external web service or (b) arbitrary Javascript that gets executed.

  • In order to get a good idea on the effectiveness of the livdewords implementation, we need to be able to track well, what users do within the modules framework. For this purpose we use Livewords instrumentation on the basis of Universal Link Tracking (ULT)

Overlay Layout

  • overlay := header(brand) panel+ ads footer
  • panel := module+

Demos

A demo page can be found here: http://industryever.corp.yahoo.com/~jmeyer/test/livewords.html

Open Issues

  • dynamically loading the annotations (cross-domain requests, json, iframes; upon load the browser needs to post one or more portions of the content to the LW platform for annotation
  • Specification for modules to include/exclude, client configuration
  • Specification for entities to detect/ignore
  • module base class API
  • usage scenarios (publisher/preannotated, publisher/non-annotated, end-user (greasemonkey/toolbar)

-- JmeyerYahoo - 25 Jan 2006
